# 精简版 Lean 说明

这份文档描述的是当前这个 **已切断 QuantConnect 鉴权和在线控制面** 的 Lean 版本。

目标不是教你怎么用 QuantConnect 服务，而是说明：

1. 这个版本还保留了什么能力
2. 哪些能力已经被禁用
3. 现在正确的开发和运行流程是什么
4. 如果你要继续做自研交易系统，下一步该怎么推进

## 当前状态

这个 Lean 版本已经做了第一阶段脱钩：

- 默认不再连接 `QuantConnect.Api.Api`
- 默认不再使用 `job-user-id` 和 `api-access-token`
- 默认不再生成 QuantConnect 鉴权链接
- 默认不再调用 QuantConnect 云项目、云回测、云优化、云 Object Store、云 live command

当前默认 API 处理器已经切成：

- `QuantConnect.Api.NoAuthApi`

它的行为是：

- 对本地运行必须保留的能力：继续工作
- 对依赖 QuantConnect 在线服务的能力：显式告警并跳过

## 保留的能力

这个精简版 Lean 目前仍然保留这些核心能力：

- 本地引擎启动
- 本地回测
- 本地订单撮合
- 本地组合、风控、指标、选股、执行模型
- 本地 `ObjectStore`
- 本地文件型历史数据读取
- 非 QuantConnect 域名的普通远程文件下载
- 你自己接入的券商适配器和数据适配器

一句话概括：

**Lean 核心回测/执行面还在，QuantConnect 在线控制面已经被切断。**

## 已禁用的能力

当前版本已经统一禁用这些能力：

- QuantConnect 云项目管理
  - `CreateProject`
  - `ReadProject`
  - `ListProjects`
  - 文件增删改查

- QuantConnect 云编译/云回测/云优化
  - `CreateCompile`
  - `CreateBacktest`
  - `ReadBacktest`
  - `EstimateOptimization`
  - `CreateOptimization`

- QuantConnect live command 控制面
  - `QCAlgorithm.Link(...)`
  - `BroadcastLiveCommand(...)`

- QuantConnect 云 Object Store
  - `GetObjectStore`
  - `SetObjectStore`
  - `DeleteObjectStore`
  - `ListObjectStore`

- QuantConnect 数据商店/下载控制面
  - `ReadDataLink`
  - `ReadDataDirectory`
  - `ReadDataPrices`
  - `DownloadData`

- QuantConnect 账户/组织/在线状态接口
  - `ReadAccount`
  - `ReadOrganization`
  - `GetAlgorithmStatus`
  - `SetAlgorithmStatus`
  - `SendStatistics`

- 任何指向 `quantconnect.com` 的下载请求

这些功能现在不是“悄悄失败”，而是：

- 写日志
- 返回 `Success = false`
- 或者直接跳过

## 现在的正确理解

这个版本的 Lean 不再是“QuantConnect 本地客户端”。

你现在应该把它理解为：

- 一个本地算法交易引擎
- 一个可继续二次开发的回测/执行内核
- 一个已经去掉默认 SaaS 控制面依赖的基础设施项目

所以后续开发重点应该从：

- “怎么登录 QuantConnect”

转成：

- “怎么把剩余的控制面、数据面、部署面替换成自己的”

## 使用流程

下面这些流程，都是面向当前这个精简版 Lean。

### 流程 1：启动本地回测

目标：验证这个精简版 Lean 仍然能正常跑本地策略。

1. 准备本地历史数据目录。
2. 修改 [Launcher/config.json](/data/data_jyz/trade/Lean/Launcher/config.json)：
   - `algorithm-type-name`
   - `algorithm-language`
   - `algorithm-location`
   - `data-folder`
3. 确认你没有依赖 QuantConnect 云项目、云编译、云回测。
4. 直接本地启动 Lean。
5. 观察日志，确认：
   - 算法成功初始化
   - 数据成功读取
   - 订单成功撮合
   - 回测成功结束

这条流程的判断标准是：

- 只要你的策略和数据都在本地，这个 Lean 应该可以跑
- 如果失败，优先检查的是数据路径、算法入口、依赖环境，而不是 QuantConnect 账号

### 流程 2：运行依赖远程公开文件的策略

目标：确认远程数据源还能工作，但不允许走 QuantConnect 域名。

1. 在策略中使用自定义数据源或远程文件源。
2. 保证数据地址不是 `quantconnect.com`。
3. 启动本地回测。
4. 如果下载成功，说明当前 `NoAuthApi` 允许普通公开 URL。
5. 如果下载地址命中了 `quantconnect.com`，当前版本会直接阻断并告警。

这条流程的设计意图是：

- 保留“普通互联网下载能力”
- 禁掉“QuantConnect 控制面和付费面”

### 流程 3：使用本地 ObjectStore

目标：确认本地持久化仍然可用。

1. 在算法里继续使用 `ObjectStore`。
2. 启动本地回测或本地 live。
3. 读写的数据会继续进入本地 `LocalObjectStore`。
4. 不要再把它理解成 QuantConnect 云存储。

正确理解是：

- `ObjectStore` 这个抽象还在
- 当前默认实现是本地磁盘
- 云 Object Store API 已经禁用

### 流程 4：策略里调用命令链接或广播命令时会发生什么

目标：明确当前禁用行为，避免误判。

1. 如果策略调用 `Link(...)`
2. 当前版本不会再生成 QuantConnect 鉴权链接
3. 会直接告警，并返回空值

另一条：

1. 如果策略调用 `BroadcastCommand(...)`
2. 当前版本不会再向 QuantConnect live command API 发请求
3. 会直接告警，并返回失败响应

所以这部分现在的定位是：

- 保留接口壳
- 禁用在线行为
- 等待后续替换成你自己的控制面

### 流程 5：本地 live 和券商接入怎么理解

目标：明确“禁用 QuantConnect 在线控制面”不等于“禁用所有 live”。

1. 先区分两层东西：
   - QuantConnect 在线控制面
   - Lean 本地 live 执行和券商适配器
2. 当前禁掉的是前者，不是后者。
3. 如果你的 live 运行只依赖本地引擎和券商 API，而不依赖 QuantConnect 云端命令/状态服务，那么理论上仍然可以继续保留。
4. 但如果某条 live 流程暗中依赖 QC 在线控制面，就会在当前版本被告警或失败。

所以对 live 的正确流程是：

1. 先做本地 paper/live 验证
2. 看失败点是在券商侧、数据侧，还是 QC 控制面侧
3. 如果是 QC 控制面侧，再做第二阶段替换

## 开发流程

下面不是使用者流程，而是你作为开发者应该怎么推进。

### 流程 6：新增自己的控制面 API

目标：把现在的 `NoAuthApi` 从“禁用壳”升级成“自研控制面入口”。

1. 先保留当前的 `NoAuthApi`
2. 把它当成过渡层
3. 逐个决定你要不要恢复这些能力：
   - 项目管理
   - 回测任务管理
   - live 控制命令
   - 状态查询
   - 远程对象存储
4. 对于你要恢复的能力，不要再接回 QuantConnect API，而是改成你自己的后端接口
5. 每恢复一类能力，就先替换一类，不要同时大改全部控制面

正确顺序建议是：

1. 先恢复 `状态上报`
2. 再恢复 `live command`
3. 再恢复 `远程对象存储`
4. 最后才考虑“项目管理/任务管理”这类外围能力

### 流程 7：继续剥离 QuantConnect 残留

目标：从“运行时断开”推进到“代码层彻底删减”。

1. 当前第一阶段已经完成：
   - 运行时默认不连 QC
   - 在线能力告警并跳过
2. 第二阶段要做的是：
   - 删除 `Api.Api`
   - 删除 `Authentication` 相关运行时代码
   - 删除 `job-user-id` / `api-access-token` 配置入口
   - 删除 QC API 对应测试
   - 删除文档和示例里的 QC 在线工作流残留
3. 第三阶段才是：
   - 把 `IApi` 缩成你真正需要的最小接口
   - 去掉现在那些只是为了兼容旧结构而保留的方法

换句话说，顺序应该是：

1. 先脱钩
2. 再裁枝
3. 最后重构接口

不要一开始就直接删到无法编译。

### 流程 8：验证当前改造是否成功

目标：验证这个版本已经从默认 QC 模式切到本地模式。

1. 检查默认 `api-handler` 是否已经是 `QuantConnect.Api.NoAuthApi`
2. 检查本地回测是否不需要任何 QC 凭证即可运行
3. 检查调用 QC 在线能力时是否出现明确告警
4. 检查访问 `quantconnect.com` 下载地址时是否被拦截
5. 检查普通公开 URL 下载是否仍然工作

如果以上都成立，说明第一阶段目标达成：

- **Lean 内核可用**
- **QuantConnect 在线控制面已被切断**

## 当前建议

你接下来最合理的推进顺序是：

1. 先用这个版本跑通一个纯本地回测
2. 再验证一个非 QuantConnect 域名的远程数据源
3. 再梳理你自己的控制面最小需求
4. 最后决定哪些 QC 遗留代码彻底删除

## 第二阶段候选任务

如果继续往下做，建议按这个顺序：

1. 删除运行时对 `Authentication.Link` 的剩余依赖
2. 删除 `Api.Api` 及其相关运行时代码
3. 删除 `job-user-id` / `api-access-token` 配置和参数解析
4. 清理所有 QC 在线测试
5. 定义你自己的最小 `IApi` 接口
6. 实现你的控制面、任务面和状态面

当前这个版本还不是最终形态。

它的定位很明确：

**它是一个已经完成“去 QuantConnect 在线依赖第一阶段”的过渡版 Lean。**

## TradingAgents 说明

这一节不是在介绍一个独立产品，而是在说明：

- `TradingAgents` 在你当前这套自研方向里扮演什么角色
- 它怎么和这个精简版 Lean 配合
- 你应该怎么用它，而不是停留在“它能分析新闻、情绪、基本面”这种表层描述

先说结论：

**TradingAgents 不应该直接替你下单。**

它在你现在这条路线里的合理定位是：

- 前端研究层
- 多源信息整理层
- 自然语言推理层
- 交易假设生成层

而不是：

- 最终执行层

执行、验证、回测、部署，仍然应该由 Lean 负责。

## TradingAgents 当前能力边界

当前这个仓库的默认工作方式是：

- 通过外部 LLM API 或本地 Ollama 跑推理
- 通过 `yfinance` 拉价格、新闻、基本面
- 通过 Reddit / StockTwits 拉情绪数据
- 最后输出多 Agent 综合结论

所以它本质上是：

- 一个研究和判断生成器
- 一个信号解释器
- 一个研究工作流编排器

不是：

- 一个可靠的交易引擎

## TradingAgents 的正确使用流程

### 流程 9：第一次跑通一轮单标的分析

目标：先验证 `TradingAgents` 能稳定给出一份完整分析，而不是一开始就和 Lean 强耦合。

1. 进入 [TradingAgents](/data/data_jyz/trade/TradingAgents) 目录。
2. 创建并激活虚拟环境。
3. 安装依赖。
4. 选择一个 LLM 提供商，或者改成本地 `ollama`。
5. 启动 CLI。
6. 只分析一只标的，例如：
   - `NVDA`
   - `SPY`
   - `QQQ`
7. 固定一个分析日期，不要一开始就多日期、多模型乱跑。
8. 等整条链跑完，确认：
   - analyst 报告完整
   - 研究团队辩论完成
   - trader 有输出
   - portfolio manager 有最终结论

这个流程的重点不是结论对不对，而是：

- 输入输出链是否完整
- 数据源是否可用
- 模型是否稳定

### 流程 10：第一次结果出来后，不直接交易，先做结构化提炼

目标：把一份自然语言分析压缩成 Lean 能验证的规则。

1. 跑完 `TradingAgents` 后，不要直接接受最终的 `buy / hold / sell` 结论。
2. 先把报告拆成这几个结构化字段：
   - 标的
   - 时间点
   - 方向
   - 触发理由
   - 风险点
   - 失效条件
3. 然后强制把它翻成一个可验证规则。

例如：

- 如果 Agent 说 `NVDA` 财报后强势延续
- 不要把这句话直接拿去实盘
- 先翻成：
  - 财报公布后的下一个交易日开盘买入
  - 持有 3 天或 5 天
  - 跌破某阈值止损

4. 只有翻译成明确规则以后，才允许进入 Lean。

这条流程的关键是：

- `TradingAgents` 输出的是“研究结论”
- Lean 需要的是“可验证规则”

中间这一步翻译，不能省。

### 流程 11：把 TradingAgents 输出送入精简版 Lean 回测

目标：建立 `研究 -> 规则 -> 回测` 的最小闭环。

1. 先选一条最简单的结论，不要一次接十几个 analyst 观点。
2. 把那条结论翻成一个最小策略规则。
3. 在精简版 Lean 里写成算法逻辑。
4. 准备对应的本地历史数据。
5. 启动本地回测。
6. 看结果时，只回答三个问题：
   - 这条规则是否有超额收益
   - 这条规则是否有可接受回撤
   - 这条规则是否稳定，而不是只在单个样本里碰巧成立
7. 如果回测不成立，不要先怀疑 Lean，先回去重新拆解 Agent 结论。

这条流程的正确顺序是：

1. Agent 给观点
2. 人把观点翻成规则
3. Lean 验证规则

不要跳过中间那步。

### 流程 12：用 TradingAgents 研究个股事件，再用 Lean 验证事件驱动策略

目标：利用 `TradingAgents` 的长处，而不是硬拿它做它不擅长的事。

它最适合的不是纯技术指标，而是这类问题：

- 财报前后市场预期和新闻情绪是否一致
- 龙头股突然大涨/大跌时，舆情和新闻是不是在强化趋势
- 大盘波动时，宏观信息是否支持短期风险偏好变化

正确流程是：

1. 选一个事件驱动场景，比如 `NVDA` 财报窗口。
2. 用 `TradingAgents` 跑事件日前后的分析。
3. 看它输出中是否真的有一致信号，而不是几类 analyst 互相冲突。
4. 把一致信号转成事件驱动规则。
5. 用精简版 Lean 在历史样本上做事件窗口回测。
6. 如果事件策略成立，再扩到更多股票和更多事件样本。

这个流程强调的是：

- 先单点验证
- 再横向扩展

不要一开始就想让 Agent 帮你跑全市场全自动策略。

### 流程 13：用 TradingAgents 做指数背景判断，再让 Lean 做组合执行

目标：把 `TradingAgents` 用在更适合它的宏观/背景判断层。

1. 先选指数，比如 `SPY` 或 `QQQ`。
2. 让 `TradingAgents` 输出这几类信息：
   - 当前宏观叙事
   - 新闻主线
   - 情绪偏向
   - 风险事件
3. 不直接把这些文本拿去交易。
4. 先把它们压缩成几个简单状态变量，比如：
   - 风险偏好偏强
   - 风险偏好偏弱
   - 宏观事件不确定性升高
5. 然后在 Lean 里把这些状态变量接入组合层逻辑：
   - 是否开仓
   - 是否降杠杆
   - 是否缩小仓位
   - 是否减少高波动股票暴露
6. 最后在回测里验证这些状态变量有没有实质提升。

这个流程里：

- `TradingAgents` 负责判断背景
- Lean 负责决定仓位、风控和执行

### 流程 14：切换模型或数据源时，怎么做对比

目标：让 `TradingAgents` 的实验具有可比性。

1. 第一轮先固定：
   - ticker
   - 日期
   - analyst 集合
   - 研究深度
   - 数据源
2. 第二轮只改一个变量。
   例如：
   - 只换模型
   - 或者只换研究深度
   - 或者只换数据 vendor
3. 对比时不要问“哪份报告写得更像样”，而要问：
   - 哪一份更容易翻译成明确规则
   - 哪一份进入 Lean 后回测更稳定
4. 如果某个模型只是更会写长文，但无法产生更可验证的规则，那对你这个系统没有实质价值。

### 流程 15：把 TradingAgents 融入你的自研系统，而不是手工反复跑 CLI

目标：从“手工试验工具”过渡到“系统组件”。

1. 第一阶段先用 CLI 跑通单次分析。
2. 第二阶段改用 Python 接口调用。
3. 把输出格式收敛成你自己的中间结构，而不是依赖最终自然语言文本。
4. 让你的系统把这些中间结构写入：
   - 文件
   - 数据库
   - 本地缓存
5. 再由下游模块把这些结构翻译成 Lean 可消费的策略输入。

正确理解是：

- `TradingAgents` 不应该成为系统的最末端
- 它应该成为上游研究模块

### 流程 16：当前这套系统里 Lean 和 TradingAgents 的分工

如果你现在开始做一套自己的交易系统，建议分工固定成这样：

1. `TradingAgents`
   - 做研究
   - 做新闻/情绪/基本面整合
   - 生成交易假设

2. 你的中间层
   - 把研究结论结构化
   - 把自然语言变成规则
   - 做信号缓存和版本化

3. 精简版 `Lean`
   - 验证规则
   - 管理策略执行逻辑
   - 管理仓位和风险
   - 做回测和 live

这个三层结构比“让 Agent 直接输出交易指令”稳得多。

## TradingAgents 和精简版 Lean 的推荐主流程

### 主流程 A：研究单股，再回测

1. 用 `TradingAgents` 分析 `NVDA`
2. 从报告中提炼一条最有信号感的判断
3. 把它翻译成明确规则
4. 在精简版 Lean 里实现
5. 用本地数据回测
6. 如果成立，再扩展到更多样本

### 主流程 B：研究指数背景，再做组合风控

1. 用 `TradingAgents` 分析 `SPY` 或 `QQQ`
2. 提炼出背景状态变量
3. 在 Lean 中把背景变量接入组合风险控制
4. 回测对组合回撤、收益、换手的影响

### 主流程 C：把 TradingAgents 变成研究组件

1. 先手工 CLI 跑通
2. 再切 Python 接口
3. 再固化成结构化输出
4. 最后挂进你的策略研究流水线

## 当前最合理的下一步

1. 先确认精简版 Lean 能稳定跑本地回测
2. 再把 `TradingAgents` 跑通一轮 `NVDA` 或 `SPY`
3. 选一条结论翻译成最小规则
4. 用 Lean 做第一次闭环验证
5. 再决定 `TradingAgents` 输出结构该怎么标准化

这一步做完以后，你的系统才算真正开始形成：

- 上游研究层
- 中间规则层
- 下游执行层
